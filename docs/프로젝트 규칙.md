# Claude Code Project Guideline (Spring / eGovFrame 스타일)

## 0) 목표
- 전자정부프레임워크(eGovFrame) 기반(스프링 MVC 구조)으로 **유지보수/가독성/테스트/확장성**이 좋은 코드를 만든다.
- SQL/프로시저 중심 개발을 고려하며, ORM(JPA) 강제하지 않는다.
- 컨트롤러는 얇게(thin) 유지하고, 핵심 로직은 Service/Domain/Repository(DAO)로 분리한다.

---

## 1) 기본 원칙 (필수)
### 1.1 계층 분리
- **Controller**: 요청/응답 매핑, 파라미터 검증, Service 호출, 응답 모델 구성만 담당
- **Service**: 비즈니스 로직의 중심, 트랜잭션 경계(가능하면 여기)
- **Repository/DAO**: DB 접근 전담 (MyBatis Mapper/DAO 또는 JDBC)
- **Domain/DTO/VO**:
  - Domain(Entity/VO): 비즈니스 의미 중심
  - DTO: 컨트롤러 입출력/외부 API 입출력 전용
  - VO: 값 객체(불변) 성격이면 VO로 분리

### 1.2 의존성 주입(DI) 규칙
- **생성자 주입을 기본값으로 사용**한다. (필드 주입 지양)
- 단일 생성자면 `@Autowired`는 생략 가능(스프링 기준).  
  (프로젝트 규칙이 있다면 일관되게 유지)

### 1.3 인터페이스 사용 규칙
- Service는 가능하면 `XxxService` 인터페이스 + `XxxServiceImpl` 구현체 구조로 간다.
- DAO/Repository도 인터페이스를 두면 테스트/교체가 쉽다.
- 컨트롤러는 구현체가 아니라 **인터페이스 타입**에 의존한다.

---

## 2) Bean 등록 방식 (우선순위)
1) 컴포넌트 스캔 방식
- `@Controller`, `@Service`, `@Repository`, `@Component`

2) Java Config의 `@Configuration + @Bean`
- 외부 라이브러리 객체 / 생성 과정이 복잡한 객체 / 조건부 생성에 사용
- `@Bean` 메서드는 반드시 `@Configuration` 클래스 안에 둔다.

3) XML 방식
- eGovFrame 레거시/기존 프로젝트 규칙상 필요한 경우만 유지

---

## 3) 트랜잭션(@Transactional) 운영 규칙 (중요)
### 3.1 어디에 붙이나?
- 기본은 **Service 메서드**에 붙인다.
- Controller에 트랜잭션 부여는 지양(예외적으로 필요한 경우만).

### 3.2 프로시저(SP)와 트랜잭션 충돌 방지
- DB 프로시저 내부에서 COMMIT/ROLLBACK을 자체 처리한다면:
  - 애플리케이션에서 동일 작업에 대해 **중복 트랜잭션을 강하게 걸지 않도록 설계**한다.
  - 원칙: 트랜잭션 주도권을 **한쪽(스프링 또는 DB)**으로 명확히 둔다.
- 혼합이 필요한 경우:
  - 트랜잭션 전파(Propagation), 격리수준(Isolation) 등을 명시적으로 설계
  - 반드시 테스트로 검증

---

## 4) SQL / MyBatis 전략 (권장)
- 복잡한 SQL(서브쿼리 중첩/가독성 저하)이 생기면:
  1) 쿼리를 분리(단계화)하거나
  2) View/CTE/임시테이블 등 DB 기능을 활용하거나
  3) 도메인/서비스 로직으로 일부 이동(단, 데이터량/성능 고려)
- “AI가 최적화해줬다”는 이유로 바로 믿지 말고,
  - 실행 계획(Explain)
  - 인덱스 설계
  - 실제 데이터 규모 기준 성능 테스트
  를 통해 검증한다.

---

## 5) AOP 적용 기준 (헬퍼와 구분)
### 5.1 헬퍼(Utility)로 푸는 게 맞는 경우
- URL 파싱, 포맷팅, 단순 계산 로직 등
- 특정 기능이 “명시적으로 호출되는 게” 더 읽기 쉬운 경우

### 5.2 AOP로 푸는 게 맞는 경우
- 모든/다수 메서드에 반복 적용되는 횡단 관심사
  - 로깅(요청/응답/실행시간)
  - 권한 체크
  - 트랜잭션(대개 스프링이 이미 AOP로 처리)
  - 감사(Audit) 기록
- AOP는 “자동으로 끼어든다”는 특성 때문에 남발 금지.
  - 적용 대상(Pointcut)을 최소화하고 문서화한다.

---

## 6) MVC 흐름 규칙 (표준)
- Request → Controller → Service → DAO(Mapper) → DB
- Response는 Controller에서 View(JSP) 또는 JSON(API)로 반환
- Controller는 “중간 다리”가 맞다. 대신 책임이 명확해야 한다:
  - 파라미터 검증/바인딩
  - 예외→응답 변환
  - 결과 모델 구성

---

## 7) 예외 처리 규칙
- 전역 예외 처리:
  - `@ControllerAdvice` + `@ExceptionHandler`로 공통 처리
- 예외는 의미 있게:
  - 비즈니스 예외(BusinessException) / 시스템 예외(SystemException) 구분
- 사용자 메시지/로그 메시지 분리:
  - 사용자에게는 안전한 메시지
  - 로그에는 원인/stacktrace

---

## 8) 보안(로그인/권한) 적용 규칙
- 가능하면 Spring Security 기반으로:
  - 인증(Authentication) / 인가(Authorization) 분리
  - URL 접근 제어 정책을 코드/설정으로 명확히 관리
- 세션 기반인지, 토큰(JWT) 기반인지 프로젝트 기준에 맞춘다.
- AI가 만든 보안 설정은 반드시 점검:
  - CSRF 정책
  - 세션 고정 보호
  - 패스워드 인코딩(BCrypt 등)
  - 권한 누락 URL 여부

---

## 9) 테스트 전략
- 단위 테스트(Unit): Service/Util 중심
- 통합 테스트(Integration): DB 연동 / Mapper 검증
- Mock 사용:
  - 외부 API, 메일, 파일 등은 Mock 처리
- 목표:
  - “리팩토링해도 깨지지 않는 안전망” 확보

---

## 10) 설정/환경 분리
- (스프링 부트가 아니라면) properties/XML 기반 환경 분리 원칙을 따른다.
- DB/외부 API/비밀키는 코드에 하드코딩 금지.
- 개발/운영 설정 분리 및 문서화.

---

## 11) 버전 관리(형상관리) 권장
- 신규/현대적 협업 기준으로는 **Git 권장**
  - 브랜치 전략: main/master + develop(선택) + feature/*
  - 커밋 메시지 규칙: feat/fix/refactor/docs/test/chore 등 prefix 권장
- 레거시/기관 요구로 SVN 강제 시:
  - trunk/branches/tags 규칙을 문서화하고 최소한의 브랜치 전략 유지

---

## 12) Claude Code에게 주는 작업 지시 템플릿
아래 형식으로 작업을 수행해라.

### 12.1 분석 단계
- 요구사항을 Controller/Service/DAO/DTO로 분해
- 트랜잭션 경계(Service)를 먼저 제안
- SQL/프로시저 호출 여부 및 책임(스프링 vs DB) 정리

### 12.2 산출물
- 패키지 구조 제안
- 핵심 클래스/인터페이스 목록
- Controller/Service/DAO 예시 코드
- 예외 처리(ControllerAdvice) 코드
- 필요한 설정 파일(스프링 설정, MyBatis 설정 등) 목록

### 12.3 절대 금지
- 컨트롤러에 비즈니스 로직 과다 작성 금지
- 필드 주입(@Autowired field) 남발 금지
- AOP 남발 금지
- DB 트랜잭션과 스프링 트랜잭션 충돌 방치 금지
- 성능 근거 없는 “최적화” 주장 금지 (테스트/플랜으로 증명)

---

## 13) 빠른 체크리스트
- [ ] Controller는 얇은가?
- [ ] Service가 트랜잭션 경계인가?
- [ ] DAO/Mapper는 DB 접근만 하는가?
- [ ] DI는 생성자 주입인가?
- [ ] 예외 처리가 전역으로 정리됐는가?
- [ ] 보안 정책(URL/권한)이 명확한가?
- [ ] 복잡 SQL은 테스트 + 실행 계획 검증 했는가?
- [ ] 설정/비밀키 분리가 되었는가?
